# Phase 2: Basic turn observation

## 1. Purpose

Observe the basic case that does not include approval first, and then solidify the basic judgment of session / message.  
At the end of this phase, we aim to be able to make advance implementation decisions without approval dependencies.

## 2. Design decisions finalized during this phase

- Adoption of `session_id = native thread ID`
- Adoption of `message_id = native item ID`
- Adoption of `turn_id` and internal usage value
- Obtainability of `event_id` and stability Primary evaluation as a contract 
- Candidates for `message.user` 
- Correspondence candidates between `message.assistant.delta` and `message.assistant.completed` 
- How to handle turn without assistant message with message projection 
- Basis for `running` 
- Judgment basis for `running -> waiting_input` 
- Should `session.status_changed` be taken directly from native runtime Primary candidate for creating 
- Basic semantics of create / start 
- Primary judgment on whether the `created` state can be safely held on the app side

## 3. Target case

Required case:

- `p2-normal-turn-complete`: Normally 1 turn completed 
- `p2-no-assistant-message`: Ends with only tool / log / request and no assistant message is output turn
- `p2-multi-turn-follow-up`: Continuation of multiple turns 
- `p2-create-start-semantics`: Check the initial behavior of create / start

This phase does not handle cases requiring approval.

## 4. Preconditions

- The observation infrastructure for Phase 1 is defined 
- Each case can be executed with the same storage format 
- The input or operation that reproduces `p2-no-assistant-message` is defined 
 If it cannot be reproduced, this phase will not mark `D. assistant no text turn confirmation` as completed.
